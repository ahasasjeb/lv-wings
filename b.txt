# 逻辑重复 / 死代码 审查报告 (谨慎版)

审查日期: 2026-05-29
范围: src/main + src/fabric + src/neoforge 全部 Java 源码（只读）
原则: 只列出“确实存在、结构上可观察到”的重复或平台分叉；不猜测运行时死代码；不激进建议删除（因为跨平台兼容需要分叉）。

## 1. 核心重复：Flights.java（最明显）
- `src/fabric/java/.../flight/Flights.java` (55 行)
- `src/neoforge/java/.../flight/Flights.java` (76 行)
- 重复内容：
  - `get(Player)`、`ifPlayer(Entity, ...)`、`ifPlayer(Entity, Predicate, BiConsumer)` 三个方法完全相同（逐字复制）。
  - onPlayerClone / onPlayerRespawn / onPlayerChangedDimension / onPlayerLoggedIn / onPlayerStartTracking 逻辑几乎一致，只是注册方式不同：
    - Fabric: 手动在 FabricServerEventHandler.register() 里注册 lambda。
    - NeoForge: @EventBusSubscriber + @SubscribeEvent 方法。
- 额外 NeoForge 特有：FLIGHT_CAPABILITY + registerCapabilities。
- 影响：维护时容易漏同步小差异（例如 onPlayerStartTracking 里 ServerPlayer 强转位置略不同）。

## 2. WingsHooks.java（平台桥接重复）
- `src/fabric/java/.../server/asm/WingsHooks.java`
- `src/neoforge/java/.../server/asm/WingsHooks.java`
- 共同：onFlightCheck(LivingEntity)、onFlightCheck(Player)、onUpdateBodyRotation、onAddFlown 签名和大部分实现相同。
- 差异：
  - Fabric 版直接调用 `FabricServerEventHandler.onXXX(ev)`。
  - NeoForge 版使用 `NeoForge.EVENT_BUS.post(ev)`。
- NeoForge 版**独有** `onGetCameraEyeHeight(Entity, float)` 方法（第30-34行）。
- 原因：Fabric 通过 mixin + WingsHooksClient 实现相机逻辑；NeoForge 把这个方法放在 server/asm 里（跨层调用）。

## 3. 客户端事件处理逻辑分叉（最大行为差异点）
- Fabric (`FabricClientEventHandler.java`):
  - END_CLIENT_TICK 里**手动循环 `client.level.players()`**，对每个 AbstractClientPlayer 调用：
    - `Flights.get(...).tick(clientPlayer)`
    - `FlightViews.get(...).ifPresent(FlightView::tick)`
  - 这意味着**所有可见玩家（含远程无翅膀玩家）在客户端都会执行飞行 tick 逻辑**。
- NeoForge (`NeoForgeClientEventHandler.java`):
  - 只订阅 `PlayerTickEvent.Post`，仅对 AbstractClientPlayer 调用 `FlightViews.get(...).tick()`（仅 view，不调用 flight.tick）。
  - flight.tick 完全依赖服务端 PlayerTickEvent（NeoForgeServerEventHandler）。
- 后果：
  - Fabric 客户端对远程玩家维护完整的 Flight 状态和动画 tracker。
  - NeoForge 客户端远程玩家 Flight 对象主要靠网络 sync 更新。
  - 可能导致细微动画/状态差异（尤其是非本地玩家）。

## 4. WingsHooksClient.java 重复 + 签名差异
- 两个独立文件（fabric/ 和 neoforge/ 目录）：
  - Fabric 版（67 行）：有 `onGetCameraEyeHeight`、`onComputeCameraRoll`、`onCheckRenderEmptyHand(5参)`、`canRenderEmptyOffhand(2参)` 等多个重载。
  - NeoForge 版（52 行）：直接 post 事件到 NeoForge.EVENT_BUS；`onCheckRenderEmptyHand` 只有 2 个参数版本；reequip 逻辑也不同（用原版 shouldCauseReequipAnimation）。
- 这些差异直接导致了 FabricCameraMixin 和 NeoForgeCameraMixin 的实现不同（见下）。

## 5. CameraMixin 重复实现
- `src/fabric/.../mixin/FabricCameraMixin.java` (66 行)：
  - 重定向 getEyeHeight + Inject alignWithEntity 手动计算 roll + 旋转 4 个向量。
- `src/neoforge/.../mixin/NeoForgeCameraMixin.java` (23 行)：
  - 仅重定向 getEyeHeight。
  - Roll 通过 `ViewportEvent.ComputeCameraAngles` 事件在 NeoForgeClientEventHandler.onCameraSetup 处理。
- 结果：NeoForge 在 server/asm/WingsHooks 里多了 onGetCameraEyeHeight 转发。

## 6. ItemInHandRendererMixin 重复 + 大差异
- Fabric 版（104 行）：有 tick() 重定向 + 5 参数 onCheckRenderEmptyHand + 额外 wings$applyEmptyOffhandSwapAnimation 逻辑。
- NeoForge 版（73 行）：简化版，只有 2 参数 onCheckRenderEmptyHand + 更少的 @Unique 字段。
- 两个 mixin 都修改同一原版类（ItemInHandRenderer），但实现细节不同，导致手持物品空手渲染行为可能不完全一致。

## 7. InSomniableCapability 重复
- Fabric 版：纯静态工具类 + 手动在 FabricServerEventHandler.COPY_FROM 里调用 onPlayerClone。
- NeoForge 版：@EventBusSubscriber + @SubscribeEvent onPlayerClone + 额外 INSOMNIABLE_CAPABILITY 注册 + registerCapabilities 方法。
- 两者都委托给 WingsAttachments，但注册/事件路径完全分叉。

## 8. Network 及消息类重复
- Network.java、MessageControlFlying.java、MessageSyncFlight.java 在 fabric/ 和 neoforge/ 各一份。
- 重复：handle 包装、clearRateLimit 转发、STREAM_CODEC 定义。
- 差异仅在发送 API（ServerPlayNetworking vs IPayloadContext / PacketDistributor）和日志。
- ControlFlyingMessageHandler 是共享的，两个 message 类只是薄 wrapper。

## 9. WingsMod 入口重复常量
- 两个 WingsMod.java 里把 WingsCore.WingSet 的 12 个翅膀常量（ANGEL_WINGS、PARROT_WINGS ... LVJIA_SUPER_WINGS）**逐字重复声明**一次。
- Names 内部类也几乎完全重复。
- 这是为了让平台代码能直接引用 `WingsMod.ANGEL_WINGS`，但造成了 20+ 行重复。

## 10. NeoForge 独有而 Fabric 缺失的功能（潜在不一致）
- NeoForgeServerEventHandler 有 `EntityMountEvent`：飞行时禁止骑乘（第48-56行）。
- Fabric 完全没有对应代码（可能通过其他 mixin 或事件实现，也可能遗漏）。
- NeoForge 有 WingsConfigEvents、PotionMix 专用类、更多 config reload 事件。
- 如果这些是必要行为，Fabric 侧存在“缺失实现”的风险（而非死代码）。

## 11. 其他小重复/胶水
- ClientFlightListenerSupport 只被两个 client proxy 调用，逻辑简单（仅本地玩家注册 notifier）。
- ControlFlyingMessageHandler 里的 LAST_CONTROL_TICKS 清理由两个平台的 message 类各自暴露 clearRateLimit。
- 没有发现真正“死代码”（被完全注释掉、从未调用的类/大块方法）。grep 未找到 TODO/FIXME/legacy 等明显标记。
- OptiFineHolder 是条件兼容代码（正常）。

## 总结（谨慎结论）
- **无明显“死代码”**（可安全删除而不影响功能的大块未使用代码）。
- **大量可观察的逻辑重复**，几乎全部源于“Fabric vs NeoForge 平台胶水”：
  - Flights、WingsHooks、WingsHooksClient、各种 Mixin、Network 消息、InSomniableCapability、WingsMod 常量。
- 最值得注意的行为分叉点：
  1. Fabric 客户端对**所有可见玩家**执行 flight.tick（NeoForge 不）。
  2. 相机 roll / 空手渲染 / 骑乘阻止 的实现路径完全不同。
- 这些重复是当前双加载器架构的**直接后果**，短期内难以消除（除非引入更多共享抽象层）。
- 建议优先级（低，因为“谨慎”）：如果未来要做统一，可考虑把 Flights/ifPlayer 等真正相同的工具方法抽到 src/main/java 的共享工具类，平台只保留注册代码。

## 12. 追加发现：功能不对称（一个加载器独有或实现差异）

### 12.1 NeoForge 独有保护：飞行时禁止骑乘
- NeoForgeServerEventHandler.java:48-56 有 `onEntityMount(EntityMountEvent)`：
  ```java
  if (event.isMounting()) {
      Flights.ifPlayer(event.getEntityMounting(), (player, flight) -> {
          if (flight.isFlying()) {
              event.setCanceled(true);
          }
      });
  }
  ```
- Fabric **完全没有**对应代码。
- 共享层 FlightListenerSupport.java:21-25 仅在“开始飞行时”调用 `player.removeVehicle()`，**不阻止**已在飞行状态时尝试 mount。
- 结论：这是**真实的功能不对称**。Fabric 上飞行玩家可能可以骑乘实体（行为不一致，可能为 bug）。

### 12.2 客户端 tick 策略差异（影响远程玩家状态）
- FabricClientEventHandler:24-34：
  - END_CLIENT_TICK 手动 `for (Player p : level.players())` 同时调用 `Flights.get(p).tick(p)` **和** `FlightView.tick()`。
  - 所有可见玩家（含远程无翅膀）都会在客户端推进飞行状态机和动画 tracker。
- NeoForgeClientEventHandler:63-68：
  - 仅在 `PlayerTickEvent.Post` 中对 AbstractClientPlayer 调用 `FlightView::tick`（不调用 flight.tick）。
  - 远程玩家飞行 tick 完全不执行，状态主要依赖网络 sync。
- 影响：Fabric 客户端远程玩家飞行动画/时间可能更“实时”；NeoForge 更依赖服务端权威同步。不是“新功能”，但会导致可观察的行为差异。

### 12.3 配置系统实现完全分叉（无共享）
- Fabric:
  - 自定义 JSON (ConfigFiles.java + WingsConfig/WingsItemsConfig 的嵌套 Data 类 + 手动 load/validate/save)。
  - 在 WingsMod.onInitialize() 一次性 validate。
  - 无热重载事件处理器。
- NeoForge:
  - Forge ModConfigSpec (toml 文件) + WingsConfig.SPEC / WingsItemsConfig.SPEC。
  - WingsConfigEvents.java 专门处理 ModConfigEvent.Loading / Reloading，调用 validate。
  - 有独立的 read* 辅助方法从 spec 同步到静态设置。
- 暴露的配置项完全相同（anti-cheat 所有参数、underwater flight、wing exertion 等），但**存储格式、热重载、默认值处理**完全不同。
- Fabric 没有 WingsConfigEvents 等效类。

### 12.4 酿造配方注册与匹配机制不同
- Fabric:
  - PotionBrewingMixin.java (3 个注入点：isIngredient / hasMix / mix)。
  - WingsBrewingRecipes.java 实现纯 Java 匹配逻辑（isSlowFalling + stream filter）。
- NeoForge:
  - PotionMix.java 继承 BrewingRecipe，重写 isInput（支持 Holder 精确匹配）。
  - 在 NeoForgeProxy:31-39 通过 RegisterBrewingRecipesEvent 注册。
- WingsBrewingCatalog 共享，实际配方一致，但实现路径和兼容性边界不同。

### 12.5 按键输入监听器实现不同
- 两者 Builder API 高度相似（支持 category + key + onPress）。
- Fabric: FabricKeyInputListener 使用 ClientTickEvents.END_CLIENT_TICK + KeyMappingHelper 注册。
- NeoForge: NeoForgeKeyInputListener 使用 @SubscribeEvent InputEvent.Key + RegisterKeyMappingsEvent + 静态列表收集。
- 功能等价（默认 R 键切换飞行），无缺失。

### 12.6 命令参数解析分叉
- FabricWingsCommand: 使用 ResourceArgument.resource(buildContext, WingsMod.WINGS_KEY) + 自定义 getWings 提取。
- NeoForgeWingsCommand: 使用自定义 WingsArgument.wings()（仅 NeoForge 有 WingsArgument.java）。
- 两者最终都委托 WingsCommandActions（共享），命令功能一致。

### 12.7 创造模式标签页填充
- 两者都有等效实现：
  - Fabric: WingsItems.register() 里用 CreativeModeTabEvents.modifyOutputEvent。
  - NeoForge: WingsItems.buildCreativeTabContents 通过 modBus 监听。
- 物品列表、顺序相同，无差异。

### 12.8 相机 Roll 实现路径差异
- Fabric: FabricCameraMixin 直接 Inject alignWithEntity，手动修改 Quaternionf + 三个 Vector3f。
- NeoForge: 通过 ViewportEvent.ComputeCameraAngles + ClientEventHandlerSupport.computeCameraRoll（带平滑 approachDegrees）。
- NeoForgeClientEventHandler 额外订阅该事件。
- 最终视觉效果一致，但代码和事件依赖不同。

### 12.9 InSomniable（梦境捕捉器）事件
- 两者都有完整实现（左键音符盒触发奖励）。
- Fabric: AttackBlockCallback + InSomniableEventHandler。
- NeoForge: PlayerInteractEvent.LeftClickBlock (LOW priority) + 相同判断逻辑。
- 功能完全一致。

### 12.10 其他观察
- 无发现 Fabric 独有而 NeoForge 缺失的核心游戏玩法功能。
- NeoForge 侧整体事件驱动更“原生”，Fabric 更多手动注册 + mixin。
- 所有“独有”代码均为平台 API 适配结果，而非业务逻辑遗漏。
- 最大实际用户可见差异风险点仍是 **12.1 的飞行骑乘保护缺失**（Fabric）。

报告结束。所有结论均基于源码静态阅读，未运行测试或反编译。
